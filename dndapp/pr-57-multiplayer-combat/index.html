PR #57 — Multiplayer Combat (Phase 9B)

Feature branch: feat/phase-9b-multiplayer-combat | Closes #49–#56

Overview

Host-authoritative multiplayer combat. The DM client resolves all actions and broadcasts full CombatState snapshots to every player. Turn enforcement ensures only the active player can act, while everyone else watches in real time.

1. Combat Initialization

When the host starts combat from the lobby, the system builds CombatUnits from each player's character snapshot, rolls initiative, and broadcasts combat_start to all clients with the full initial state.

📸 Screenshot: Combat starting with initiative order showing all players' characters

2. Turn Enforcement — Active Player

Only the player whose turn it is can interact with the combat UI. Their action panel is fully enabled while other players see a "Waiting for [name]'s turn" state.

📸 Screenshot: Active player's turn with enabled action buttons

3. Player Command Protocol

12 typed command variants: move, attack, castSpell, dash, dodge, disengage, endTurn, useConsumable, useClassFeature, useReaction, passReaction, selectWeapon, helpAction. Commands are validated on the host before resolution.

📸 Screenshot: Player sending a command, host resolving it

4. Real-Time State Sync

After every action, the host broadcasts a serialized CombatState snapshot with a monotonic sequence number. Stale messages are rejected, ensuring all clients stay in sync.

📸 Screenshot: Two browser tabs showing synchronized combat state

5. Multiplayer Reactions

When a reaction trigger occurs (e.g., Shield spell), the host sends a reaction_request to the owning player. They have 30 seconds to respond before auto-pass kicks in.

📸 Screenshot: Reaction prompt appearing on the correct player's screen

6. DM Controls

The host/DM controls all enemy units via the DmActionPanel. Their actions resolve locally and broadcast to all players. Zero impact on solo play — all multiplayer paths are gated behind sessionStatus === 'combat'.

📸 Screenshot: DM controlling enemy units while players watch

Technical Highlights

  • PlayerCommand protocol — 12 typed command variants with host-side validation
  • SerializedCombatState — Full snapshots with sequence numbers for ordering
  • useMultiplayerCombat hook — Turn enforcement + dispatch routing (solo → store, multiplayer → broadcast)
  • Reaction routing — Targeted reaction_request with 30s timeout + auto-pass
  • State persistence — JSONB snapshots at turn boundaries for reconnect recovery
