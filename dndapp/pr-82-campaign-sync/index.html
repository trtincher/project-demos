Campaign Cloud Sync

PR #82 · Closes #82 · Infrastructure — no visual changes

The Problem

Campaigns were local-only. Unlike characters and custom encounters, they had no Supabase sync — if you cleared browser data or logged in from another device, your campaigns were gone. The campaignStore comment claimed to follow "the same dual-write pattern as characterStore" but only the Dexie half had been implemented.

Before

  • Campaigns in IndexedDB only
  • Lost on browser data clear
  • Not available across devices
  • 12 campaign store actions, 0 enqueueSync calls

After

  • Dual-write: Dexie → Supabase queue
  • Last-write-wins via updated_at
  • Pull-from-cloud on sign-in
  • All 12 actions queue cloud sync

Schema

New migration 20260405000000_campaigns.sql mirrors the client shape:

CREATE TABLE public.campaigns (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  name TEXT NOT NULL,
  description TEXT NOT NULL DEFAULT '',
  encounters JSONB NOT NULL DEFAULT '[]'::jsonb,
  party_character_ids TEXT[] NOT NULL DEFAULT '{}',
  current_encounter_index INTEGER NOT NULL DEFAULT 0,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

Standard RLS policies (auth.uid() = user_id) for all four CRUD operations, plus an updated_at trigger that reuses the function from the initial schema.

Ordering Note

Campaigns reference character IDs via party_character_ids. If pulled before characters on a new device, those references would dangle. pullFromCloud already pulls characters first, and first-login migration pushes characters before campaigns, so references resolve correctly on every sync path.

Conflict Resolution

Last-write-wins keyed on updated_at, matching the existing pattern for characters. Concurrent edits from two devices will clobber one another — acceptable given single-user DM workflows. A finer-grained merge (per-encounter) is future work if it becomes a real problem.

Scope Boundary

This is independent of the multiplayer combat_sessions system, which has its own Supabase tables and realtime channels. Campaigns are persistent DM-owned data; combat sessions are ephemeral shared state.
