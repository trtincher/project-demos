Pipeline Overview

Until now, every Supabase SQL migration in supabase/migrations/ has been applied by hand in the dashboard — fragile, easy to forget, no PR-time validation. This PR adds a GitHub Actions workflow that validates on PR and applies on merge, with a secret-gate so the workflow can land before credentials are wired up.

Secret-Gate Behavior

This is the deliberate design choice that makes this PR safe to merge immediately: if secrets aren't set yet, every substantive step is skipped and the job exits green with a warning annotation. No blocking red X on the merge queue, just a visible reminder.

Rollout Order

  1. Merge this PR. The workflow lands, but nothing runs against prod yet (secret-gate skips everything).
  2. Set three repo secrets via gh secret set: SUPABASE_ACCESS_TOKEN, SUPABASE_DB_PASSWORD, SUPABASE_PROJECT_REF.
  3. Trigger the first real run: gh workflow run "Supabase migrations". This applies the outstanding 20260405000000_campaigns.sql from #83.
  4. Verify: campaigns table exists in Supabase, RLS policies active, smoke test cross-device sync.
  5. From then on, any migration added to supabase/migrations/ applies automatically on merge.
Why not Option A (Supabase native GitHub integration)? It's faster to set up (~10 min) but the config lives in the Supabase dashboard, invisible to anyone reading the repo. Option B (this PR) costs an extra 20 minutes and buys: explicit workflow source, PR-time dry-run validation, GitHub-native audit trail, and loud failure modes if something drifts.
